# Zen-Gamer Платформын Системийн Архитектур болон Логик
*Дипломын төсөлд зориулсан техникийн дэлгэрэнгүй баримт бичиг*

---

## 1. Системийн Ерөнхий Архитектур (System Overview)

Zen-Gamer нь цахим спортын тамирчид, дасгалжуулагч болон энгийн тоглогчдыг холбосон маркетплейс платформ юм. Систем нь **Client-Server** архитектур дээр суурилсан бөгөөд дараах технологийн стек дээр ажиллана:

- **Frontend**: React.js (Vite), Tailwind CSS, Framer Motion (UI анимаци)
- **Backend**: Express.js (Node.js) - Vercel Serverless функц хэлбэрээр ажиллана.
- **Өгөгдлийн сан**: NeonDB (Serverless PostgreSQL)
- **Файл хадгалалт**: Cloudinary (Зураг, медиа файл)
- **Хиймэл оюун (AI)**: Google Gemini API (Ухаалаг туслах)

---

## 1.1 Архитектурын Хэв Шинж: Монолит уу, Микросервис үү?

Дипломын ажлын хүрээнд системийн архитектурын хэв шинжийг тодорхойлох нь маш чухал юм. Zen-Gamer платформ нь **Decoupled Monolithic Backend API & Serverless SPA** (Салгасан Монолит Бэкенд ба Серверлэг SPA) архитектурын хэв шинжийг агуулдаг.

### 1. Фронтенд болон Бэкендийн Тусгаарлалт (Decoupled Client-Server)
Фронтенд (React програм) болон Бэкенд (Express API) хоёр нь хоорондоо бүрэн хамааралгүй бие даасан хоёр систем юм. Тэд зөвхөн HTTP REST API-аар дамжуулан харилцана. Энэ нь:
- Фронтендийг ямар ч сервергүйгээр Статик байдлаар (Vercel, Netlify) хурдан деплой хийх боломж олгоно.
- Бэкендийг өөрчилж байх явцад фронтенд тасалдахгүй ажиллах нөхцөлийг бүрдүүлнэ.

### 2. Бэкендийн Монолит Шинж чанар (Monolithic API)
Бэкенд нь **Монолит** (Monolithic) бүтэцтэй. Системийн бүх бизнес логик (Хэрэглэгчид, Бараа бүтээгдэхүүн, Захиалга, Дундын данс/Escrow, Чат, Мэдэгдэл) нь нэг кодны сан (`api/index.mjs`) дотор байрлаж, нэг өгөгдлийн санг (NeonDB) дундаа ашигладаг.
- **Яагаад микросервис сонгоогүй вэ?**: Микросервис архитектур нь маш өндөр нарийн төвөгтэй сүлжээний тохиргоо, үйлчилгээ хоорондын харилцан хамаарал, өгөгдлийн зөрчилдөөнийг бий болгодог. Одоогийн академик болон дунд хэмжээний системд нэгдсэн Монолит API ашиглах нь хурд, найдвартай байдал, гүйлгээний бүрэн бүтэн байдлыг (Database Transactions) хангахад хамгийн тохиромжтой сонголт байсан.

### 3. Сервергүй болон Төлөвгүй Шинж (Stateless Serverless Backend)
Хэдийгээр бэкенд логик нь монолит боловч дэд бүтцийн хувьд **Serverless** (Сервергүй) хэлбэрээр Vercel дээр ажилладаг.
- Сервер нь байнга асаалттай байхгүй бөгөөд хэрэглэгч хүсэлт илгээх үед Vercel Serverless Function ажиллаж, хүсэлтийг шийдвэрлээд буцаж унтардаг.
- Энэ нь системийн төлөвгүй (Stateless) байхыг шаарддаг бөгөөд сешн болон төлөвүүдийг серверийн санах ойд биш, зөвхөн өгөгдлийн сан (NeonDB) болон JWT токен дотор хадгалж ажилладаг.

## 2. Өгөгдлийн Сангийн Бүтэц (Database Schema)

Системийн өгөгдөл нь хоорондоо хамааралтай (Relational) дараах хүснэгтүүдээс бүрдэнэ:

### 2.1 `users` Хүснэгт (Хэрэглэгчид)
Хэрэглэгчийн бүртгэл, профайл болон санхүүгийн үлдэгдлийг хадгална.
- `id` (TEXT, PK): Хэрэглэгчийн давтагдашгүй UUID
- `username` (TEXT, Unique): Хэрэглэгчийн нэр
- `email` (TEXT, Unique): И-мэйл хаяг
- `password` (TEXT): Шифрлэсэн нууц үг (bcrypt hash)
- `avatar` (TEXT): Профайл зургийн URL хаяг
- `rank` (TEXT): Эрхийн түвшин (`OPERATOR`, `moderator`, `admin`, `owner`, `BANNED`)
- `bio` (TEXT): Өөрийн тухай товч мэдээлэл
- `balance` (REAL): Хэтэвчний үлдэгдэл (₮-өөр)
- `created_at` (TIMESTAMPTZ): Бүртгүүлсэн огноо

### 2.2 `products` Хүснэгт (Бүтээгдэхүүн/Үйлчилгээ)
Хэрэглэгчдийн зарж буй цахим спорттой холбоотой үйлчилгээ болон техник хангамжууд.
- `id` (TEXT, PK): UUID
- `user_id` (TEXT, FK): Зарж буй хэрэглэгчийн ID (`users.id`)
- `title` (TEXT): Зарлалын гарчиг
- `description` (TEXT): Дэлгэрэнгүй тайлбар
- `category` (TEXT): Ангилал (`BOOSTING`, `COACHING`, `GEAR`, `SUPPLEMENTS`)
- `wear_condition` (TEXT): Барааны элэгдлийн төлөв (Жишээ нь: Шинэ, Шинэвтэр, Хуучин)
- `region` (TEXT): Бүс нутаг (`GLOBAL`, `ASIA`, `EUROPE` г.м)
- `basic_price`, `pro_price`, `elite_price` (REAL): Гурван төрлийн багцын үнэ
- `basic_name`, `pro_name`, `elite_name` (TEXT): Багцуудын нэршил
- `images` (JSONB): Зургуудын URL хаягуудын жагсаалт
- `status` (TEXT): Идэвхтэй эсэх (`ACTIVE`, `DELETED`)

### 2.3 `orders` Хүснэгт (Худалдан Авалт)
Хэрэглэгчдийн хийсэн ерөнхий захиалгуудыг бүртгэх хүснэгт.
- `id` (TEXT, PK): UUID
- `buyer_id` (TEXT, FK): Худалдан авагчийн ID (`users.id`)
- `product_id` (TEXT, FK): Барааны ID (`products.id`)
- `tier` (TEXT): Сонгосон багц (`BASIC`, `PRO`, `ELITE`)
- `status` (TEXT): Захиалгын төлөв (`PENDING`, `COMPLETED`, `CANCELLED`)
- `total` (REAL): Нийт дүн
- `created_at` (TIMESTAMPTZ): Үүссэн огноо

### 2.4 `escrow_trades` Хүснэгт (Дундын Дансны Баталгаажуулалт)
Тоглоомын аккаунт болон дижитал хөрөнгө солилцох аюулгүй байдлыг хангах дундын дансны систем.
- `id` (TEXT, PK): UUID
- `buyer_id` (TEXT, FK): Худалдан авагч (`users.id`)
- `seller_id` (TEXT, FK): Борлуулагч (`users.id`)
- `product_id` (TEXT, FK): Солилцож буй бүтээгдэхүүн (`products.id`)
- `amount` (REAL): Дундын дансанд түгжсэн мөнгөн дүн
- `account_credentials` (TEXT): Борлуулагчийн оруулсан аккаунтын мэдээлэл (Нэвтрэх нэр, нууц үг)
- `status` (TEXT): Гүйлгээний явцын төлөв (`PENDING_SELLER_CREDS`, `PENDING_MIDDLEMAN_VERIFICATION`, `COMPLETED`, `CANCELLED`)
- `middleman_id` (TEXT): Шалгасан админ эсвэл модераторын ID

### 2.5 `conversations` болон `messages` Хүснэгтүүд (Чат)
Хэрэглэгч хоорондын чат, мессежний систем.
- `conversations.id` (TEXT, PK): UUID
- `conversations.buyer_id` / `seller_id` (TEXT, FK): Холбоотой хэрэглэгчид
- `messages.id` (TEXT, PK): UUID
- `messages.conversation_id` (TEXT, FK): Аль чатны зурвас болох
- `messages.sender_id` (TEXT, FK): Зурвас илгээгч хэрэглэгч
- `messages.text` (TEXT): Зурвасны агуулга
- `messages.read` (BOOLEAN): Уншсан эсэх

## 3. Баталгаажуулалт болон Эрхийн Удирдлага (Auth & Authorization)

Хэрэглэгч системд нэвтрэх үед **JWT (JSON Web Token)** ашиглан баталгаажуулалт хийгдэнэ.

1. **Бүртгэл (Register)**: Хэрэглэгчийн оруулсан нууц үгийг шууд хадгалахгүй, `bcrypt` ашиглан 10 удаагийн давтамжтайгаар шифрлэж (hash) өгөгдлийн санд аюулгүй хадгална.
2. **Нэвтрэх (Login)**: Хэрэглэгчийн оруулсан и-мэйл болон нууц үгийг баазад байгаа шифрлэсэн нууц үгтэй `bcrypt.compare` ашиглан харьцуулна. Зөв бол систем хэрэглэгчийн ID, нэр болон эрхийн түвшинг агуулсан JWT токен үүсгэж буцаана.
3. **Frontend талын хамгаалалт**: Буцаж ирсэн токенийг `localStorage`-д хадгалж, API руу хийж буй хүсэлт бүрийн `Authorization: Bearer <token>` толгой (header) хэсэгт автоматаар хавсаргаж илгээнэ.
4. **Эрхийн шатлал (Role-based access controls)**:
   - `OPERATOR`: Энгийн хэрэглэгч (Бараа зарах, худалдан авах, чатлах).
   - `moderator`: Дундын дансны гүйлгээг хянаж, дансны мэдээллийг шалгах эрхтэй.
   - `admin`: Хэрэглэгчдийг хориглох (бандох), админы тоймыг харах, гүйлгээ удирдах эрхтэй.
   - `owner`: Системийн үүсгэн байгуулагч. Бусад админуудын эрхийг өөрчлөх дээд эрхтэй.
   - `BANNED`: Систем рүү нэвтрэх болон ямар нэгэн үйлдэл хийх эрхгүй.

## 4. Үндсэн Бизнес Логикууд (Core Business Logic)

### 4.1 Маркетплейс болон Бараа Байршуулах Логик
- Хэрэглэгч шинээр үйлчилгээ эсвэл бүтээгдэхүүн нэмэх үед зураг оруулахад `Cloudinary API`-ийг дуудаж, зургийг Cloud сүлжээнд хадгалан буцаж ирсэн найдвартай URL хаягуудыг `products.images` (JSONB) баганад хадгална.
- Бараа захиалах үед тухайн барааны үнэ `BASIC`, `PRO`, `ELITE` сонголтоос хамаарч динамикаар тооцогдоно.

### 4.2 Дундын Дансны Баталгаажуулалт (Escrow System)
Энэ нь цахим спортын аккаунт, тоглоомын кодуудыг залилуулахгүйгээр аюулгүй арилжаалахад зориулагдсан хамгийн чухал логик юм:

```
[Худалдан авагч]               [Систем / Дундын данс]               [Борлуулагч]
      |                                 |                                 |
      |--- 1. Төлбөр хийх -------------->|                                 |
      |    (Мөнгө системд түгжигдэнэ)   |                                 |
      |                                 |--- 2. Мэдэгдэл илгээнэ -------->|
      |                                 |                                 |
      |                                 |<-- 3. Аккаунтын мэдээлэл илгээнэ|
      |                                 |                                 |
      |                                 | === 4. АДМИН ШАЛГАЛТ ХИЙНЭ ===  |
      |                                 |                                 |
      |                                 |-- [ЗӨВ БОЛ] --------------------> [Төлбөр шилжинэ]
      |<-- [Аккаунтын мэдээлэл авна] ---|
      |                                 |
      |                                 |-- [БУРУУ БОЛ] ------------------> [Буцаалт хийгдэнэ]
      |<-- [Төлбөр буцаж орно] ---------|
```

1. **Төлбөр Түгжих (Create Escrow)**: Худалдан авагчийн хэтэвчнээс захиалгын мөнгөн дүн хасагдаж, `escrow_trades` хүснэгтэд `PENDING_SELLER_CREDS` төлөвтэйгөөр түр хадгалагдана.
2. **Мэдээлэл Илгээх (Submit Credentials)**: Борлуулагч системд аккаунтын нэвтрэх мэдээллээ оруулж илгээснээр төлөв `PENDING_MIDDLEMAN_VERIFICATION` болно.
3. **Админы Хяналт (Verify / Cancel)**: Админ эсвэл модератор мэдээллийг өөрийн биеэр шалгаад:
   - **Баталгаажуулах (Verify)**: Зөв байвал төлбөрийг борлуулагчийн баланс руу шилжүүлж, худалдан авагчид аккаунтын мэдээллийг илгээнэ. Төлөв `COMPLETED` болно.
   - **Цуцлах (Cancel)**: Буруу байвал мөнгийг худалдан авагчийн баланс руу буцаан олгоно. Төлөв `CANCELLED` болно.

### 4.3 Ухаалаг Чат болон Gemini AI Туслах
- Хэрэглэгчийн мессежний хэсэгт систем доторх хэрэглэгчид шууд чатлах боломжтойгоос гадна хиймэл оюунтай харилцах боломжтой.
- Бэкенд дээр байрлах `/api/gemini` болон `/api/gemini-nano` proxy хаягуудаар дамжуулан Google-ийн `gemini-2.0-flash` загварыг холбож, хэрэглэгчдэд цахим спорт, тоглоомын зөвлөгөө өгөх, тусламжийн системээр хангах үйлчилгээг бодит хугацаанд (SSE - Server Sent Events) урсгал (stream) байдлаар үзүүлдэг.

## 5. Фронтенд Архитектур (Frontend Architecture)

- **State Management**: React-ийн өөрийн `useState`, `useEffect` hook-үүдийг ашиглан хуудасны төлөвүүдийг удирдсан.
- **Хуудас Шилжилт (Routing)**: `react-router-dom` ашиглан `App.tsx` дээр бүх хуудсуудын замыг зааж өгсөн. `AnimatePresence` болон `PageTransition` компонентуудыг ашиглаж хуудас солигдох үед cyberpunk загвартай зөөлөн хөдөлгөөнүүдийг оруулсан.
- **API Wrapper**: Бүх серверийн хүсэлтүүд нь `src/lib/api.ts` файлд модульчлагдсан байдлаар бичигдсэн нь кодыг дахин ашиглах болон удирдах боломжийг сайжруулсан.

### 5.1 Фронтенд болон Бэкенд Хоорондын Холболт (Fetch API vs Axios)

Програмд гадны сангуудаас (жишээ нь `axios`) хамааралтай байдлыг багасгаж, програмын ачаалах хурдыг нэмэгдүүлэх зорилгоор хөтөчийн стандарт **`Fetch API`** технологийг сонгон ашигласан.

1.  **Төвлөрсөн Хүсэлтийн Модуль (`request` wrapper)**:
    `src/lib/api.ts` файл дотор `request<T>()` нэртэй ерөнхий wrapper функцийг үүсгэсэн. Энэ нь дараах үүрэгтэй:
    *   **Автомат Баталгаажуулалт (JWT Auto-attach)**: `localStorage`-д хадгалагдсан JWT токенийг уншиж, хэрэв токен байвал API руу явах хүсэлт болгоны Header-т `Authorization: Bearer <token>` хэлбэрээр автоматаар хавсаргана.
    *   **Өгөгдлийн Төрөл Тохируулах (Content-Type)**: Илгээж буй өгөгдлийн бүтцийг шалгаж, ерөнхий JSON төрөл бол `Content-Type: application/json` болгож хөрвүүлнэ. Харин зураг оруулах үед (Cloudinary руу илгээхэд) `FormData` бүтцийг ашиглан файл хэлбэрээр дамжуулна.
    *   **Төвлөрсөн Алдаа Барилт (Centralized Error Handling)**: Хэрэв серверээс `res.ok` бус буюу `4xx` эсвэл `5xx` төрлийн алдаа ирвэл хариуг JSON хэлбэрээр уншиж, серверийн буцаасан алдааны мессежийг (`data.error`) шууд JavaScript-ийн шинэ `Error` объект болгон шиднэ (`throw new Error`). Үүнийг нь фронтендын React компонентууд `try-catch` ашиглан хялбархан барьж авах боломжтой.

2.  **Fetch API-ийг Сонгосон Шалтгаан**:
    *   **Хамааралгүй байдал (No Third-Party Overhead)**: Гадны том санг (`axios` гэх мэт) төсөлд нэмж суулгах шаардлагагүй тул бэлэн болсон бэлдэцийн (Build) хэмжээ бага байна.
    *   **Урсгал хариу авах боломж (SSE Streaming)**: Google Gemini AI-аас мессежний хариуг бодит хугацаанд үг үсгээр нь урсгаж авахын тулд стандарт `fetch` нь `ReadableStream` болон Server-Sent Events-тэй илүү хялбар хоршиж ажилладаг.

## 6. Баримт Бичгийг PDF Болгож Хөрвүүлэх Заавар

Энэхүү баримт бичгийг дипломын ажилдаа хавсаргахын тулд дараах хялбар аргуудын аль нэгээр PDF болгон хөрвүүлээрэй:

### Арга 1: VS Code ашиглах (Хамгийн хялбар бөгөөд чанартай)
1. VS Code дээрээ `Markdown PDF` эсвэл `Markdown All in One` өргөтгөлийг (Extension) суулгана.
2. Энэхүү `diploma_logic_documentation.md` файлыг нээнэ.
3. Хулганы баруун товчийг дараад **`Markdown PDF: Export (pdf)`** командыг сонгоно.
4. Файлын хажууд бэлэн PDF файл үүсэх болно.

### Арга 2: Браузер ашиглах
1. Энэхүү markdown файлыг Github эсвэл ямар нэгэн Markdown үзүүлэгч (Viewer) дээр нээнэ.
2. Гарны `Ctrl + P` (Хэвлэх) товчийг дарна.
3. Destination хэсгээс **`Save as PDF`** сонголтыг сонгоод хадгална.

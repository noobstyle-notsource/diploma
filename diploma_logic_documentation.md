## 1-р Бүлэг: ТАНИЛЦУУЛГА (15 Оноо)

### 1.1 Төслийн зорилго
Манай орны цахим спортын (Esports) хурдацтай хөгжлийг даган тоглоомын дижитал хөрөнгө (аккаунт, арьс/skin, тоглоомын код) арилжаалах, рэнк ахиулах үйлчилгээ авах, мэргэжлийн дасгалжуулагч олох эрэлт хэрэгцээ асар их нэмэгдсэн. Энэхүү төслийн зорилго нь цахим спортын тамирчид, дасгалжуулагч болон сонирхогчдыг нэг дор цуглуулж, ямар нэгэн залилангийн эрсдэлгүйгээр худалдаа хийх боломжтой **аюулгүй дундын дансны систем (Escrow) бүхий Zen-Gamer платформыг** хөгжүүлэхэд оршино.

### 1.2 Төслийн зорилт ба хамрах хүрээ
*   **Зорилт 1**: Хэрэглэгчид өөрсдийн үйлчилгээ болон техник хангамжийг хялбар байршуулах систем бүтээх.
*   **Зорилт 2**: Бараа болон үйлчилгээг худалдан авагч талд хохиролгүй дамжуулах админ/модераторын баталгаажуулалт бүхий дундын дансны ухаалаг логикийг (Middleman Escrow) нэвтрүүлэх.
*   **Зорилт 3**: Худалдан авагч болон борлуулагч хооронд шууд харилцах чатын систем үүсгэхээс гадна Google Gemini AI ухаалаг зөвлөхийг системд нэгтгэх.
*   **Хамрах хүрээ**: Монгол улсын цахим спорт сонирхогч залуучууд, мэргэжлийн болон хагас мэргэжлийн тоглогчид, цахим спортын чиглэлийн техник хэрэгслийн худалдаа эрхлэгчид.

## 2-р Бүлэг: АСУУДАЛ БА ШИЙДЭЛ (25 Оноо)

### 2.1 Асуудлыг нотолгоотой тодорхойлсон байдал
Одоогийн байдлаар Монгол тоглогчид Facebook групп, Discord серверүүдээр дамжуулан тоглоомын аккаунт болон кодоо арилжаалж байна. Энэ нь дараах томоохон асуудлуудыг дагуулдаг:
1.  **Залилан (Scam)**: Гүйлгээ хийсний дараа борлуулагч нь аккаунтаа буцааж авах эсвэл худалдан авагч мөнгөө шилжүүлэхгүй блок хийж зугтах тохиолдол өдөр тутам хэдэн арваараа гарч байна.
2.  **Хуурамч дундын админ (Fake Middleman)**: Фэйсбүүк группүүдэд нэр хүнд бүхий админы нэр, зурагтай хуурамч хаяг нээн хэрэглэгчдийг залилах асуудал маш өндөр байна.
3.  **Тогтворгүй үнэ**: Нэгдсэн систем байхгүйгээс үйлчилгээ болон барааны ханш зах зээлийн бодит үнэлгээнээс хэт зөрүүтэй байдаг.

### 2.2 Шийдлийн арга зам болон түүний үндэслэл
Zen-Gamer нь эдгээр асуудлыг системчилсэн **Дундын дансны (Escrow) гүйлгээний урсгал**-аар шийдсэн. Энд ямар нэгэн фэйсбүүк чат ашиглахгүй бөгөөд зөвхөн системд баталгаажсан админ/модераторууд гүйлгээ бүрийг бие даан шалгасны үндсэн дээр мөнгийг суллах эсвэл буцаан олгох үйлдлийг хийнэ.

### 2.3 Өрсөлдөгч системүүдийн харьцуулалт
| Харьцуулах үзүүлэлт | Facebook Group-үүд | Гадны систем (G2G, PlayerAuctions) | Zen-Gamer Платформ |
| :--- | :--- | :--- | :--- |
| **Аюулгүй байдал** | Маш бага (0%) | Өндөр | **Маш өндөр (Escrow + Admin)** |
| **Шимтгэл** | Бага эсвэл үнэгүй | Өндөр (10%-15% + Ханшны зөрүү) | **Дундаж / Бага** |
| **Монгол Төгрөгөөр (MNT)** | Боломжтой боловч эрсдэлтэй | Боломжгүй (Зөвхөн USD/EUR) | **Бүрэн дэмжсэн (MNT)** |
| **Харилцагчийн туслах** | Байхгүй | Удаан, Монголоор ярьдаггүй | **Бодит хугацааны AI (Gemini) + Монголоор** |

## 3-р Бүлэг: ТЕХНИКИЙН ДЭД БҮТЭЦ (20 Оноо)

### 3.1 Технологийн сонголтын үндэслэл
*   **React.js (Frontend)**: SPA (Single Page Application) хөгжүүлэхэд хамгийн тохиромжтой, хэрэглэгчийн үйлдэл бүрт хуудсыг дахин ачаалах шаардлагагүй тул гүйцэтгэл өндөр.
*   **Tailwind CSS**: Custom загвар болон Cyberpunk дизайны хэв маягийг уян хатан, хурдан кодлох боломжийг олгосон.
*   **Express.js (Backend)**: Node.js орчны хамгийн хөнгөн бөгөөд ажиллагаа хурдтай вэб фреймворк. Vercel Serverless орчинд хамгийн бага нөөцөөр ажиллах чадвартай.
*   **NeonDB (PostgreSQL)**: PostgreSQL-ийн серверлэг (Serverless) хувилбар бөгөөд өгөгдлийн сангийн найдвартай ажиллагаа болон relational (холбоост) аюулгүй байдлыг 100% хангана.

### 3.2 Архитектурын зөв зохион байгуулалт
Систем нь **Decoupled Monolithic Backend API & Serverless SPA** бүтэцтэй. 
*   **Stateless API**: Сервер нь Vercel дээр ажиллах тул ямар нэгэн төлөв (State) өөртөө хадгалахгүй. Энэ нь серверийн ачааллыг эрс багасгаж, өгөгдлийн зөрчил гарах эрсдлийг тэглэнэ.
*   **Fetch API Wrapper**: Axios гэх мэт том хэмжээний сангуудын оронд хөтөчийн стандарт `Fetch API`-ийг төвлөрүүлж ашигласнаар програмын хэмжээг маш хөнгөн (dist хавтасны хэмжээ ердөө 600kB орчим) болгосон.

### 3.3 Аюулгүй байдал болон Гүйцэтгэл
1.  **JWT Баталгаажуулалт**: Бүх хүсэлтийг токеноор баталгаажуулах тул хэрэглэгчийн нэвтрэх сешн дуусах хүртэл аюулгүй хадгалагдана.
2.  **Bcrypt Нууцлал**: Нууц үгийг баазад `bcrypt` ашиглан шифрлэж хадгалдаг тул өгөгдлийн сан алдагдсан ч нууц үг тайлагдах боломжгүй.
3.  **CORS & Environment Variables**: Бэкенд сүлжээний аюулгүй байдлыг CORS-оор хамгаалж, чухал түлхүүрүүдийг (Gemini API Key, DB URL, Cloudinary Secret) Vercel орчны нууц хувьсагчид хадгалсан.

## 4-р Бүлэг: КОДЫН ХЭРЭГЖИЛТ (25 Оноо)

### 4.1 Кодын бүтэц (Code Organization)
Төслийн кодын зохион байгуулалт нь эмх цэгцтэй, модульчлагдсан хэлбэртэй:
```
shantralt1.2/
├── api/                  # Backend: Express Serverless API
│   └── index.mjs         # API Route болон Controllers
├── src/                  # Frontend: React Application
│   ├── components/       # Дахин ашиглагдах UI Компонентууд (Navbar, Footer, Bell)
│   ├── lib/              # Холболтын хэсэг (api.ts - Fetch wrapper)
│   ├── pages/            # Хуудсууд (Home, Products, Admin, Escrow)
│   ├── App.tsx           # Router болон Routes
│   └── main.tsx          # Эхлэл цэг
```

### 4.2 Гол функциональ шаардлагын хэрэгжилт
*   **Автомат Хүсэлт ба JWT**: `src/lib/api.ts` доторх `request` функц нь хэрэглэгчийн токенийг `Authorization` header-т автоматаар хавсаргана.
*   **Дундын дансны модуль**: Хэрэглэгчдийн хоорондын дижитал гүйлгээ модераторын баталгаажуулалтаар ажиллах логик бэкенд болон админ самбарт бүрэн хэрэгжсэн.
*   **Идэвхтэй Чат ба Gemini AI**: Хэрэглэгчийн туслахад Gemini загварыг холбож, ухаалаг зөвлөх систем бодит хугацаанд ажиллаж байна.

### 4.3 Тестлэлт ба Демо ажиллагаа
Төслийг локал орчинд болон Vercel-ийн CI/CD дээр амжилттай build хийсэн:
*   **Локал тест**: `npm run build` командаар кодыг шалгахад амжилттай хөрвүүлэгдсэн.
*   **Демо хаяг**: Платформыг Vercel дээр байрлуулж, өгөгдлийн санг NeonDB-тэй холбосноор демо ажиллагаа бүрэн хангагдсан.
